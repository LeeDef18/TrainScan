## 1. Цели и предпосылки

### 1.1. Зачем идем в разработку продукта?

**Бизнес-цель (Product Owner)**
Стратегическая цель - повышение операционной эффективности диспетчерской службы за счет автоматизации процесса идентификации ориентации подвижного состава. 


Внедрение ML-решения позволит:
- Снижение OPEX на 85% за счет автоматизации ручного труда диспетчеров
- Повышение качества мониторинга за счет снижения ошибок идентификации с 7.3% до <1%, что минимизирует риски аварий и необоснованных ремонтов.
- Масштабирование без роста затрат — подключение новых полигонов без найма персонала, обеспечение 24/7 мониторинга без сменной работы.

**Почему станет лучше, чем сейчас (Product Owner & Data Scientist)**


*Текущее состояние:*
- Ручной анализ видеозаписей операторами
- Субъективность оценки при плохой видимости
- Отсутствие единой базы данных ориентации вагонов
- Невозможность масштабирования без увеличения штата

*Целевое состояние:*
- Автоматическая детекция референтных объектов в реальном времени
- Объективная оценка с доверительной вероятностью >95%
- Интеграция с существующей системой мониторинга через REST API
- Масштабируемость на любой полигон без дополнительных затрат

**Критерии успеха итерации (Product Owner)**
1. Достижение точности определения ориентации вагона ≥ 92% на валидационной выборке
2. Время отработки модели ≤ 150 мс на кадр


### 1.2. Бизнес-требования и ограничения

**Бизнес-требования**
- Система должна определять ориентацию вагона с точностью ≥90% 
- Время обработки одного вагона ≤ 30 сек 
- Работа в условиях ограниченной видимости (ночь, осадки)
- Интеграция с АСУ "Транс-Телематика" через API


**Бизнес-ограничения**
1. **Технические:**
   - Частота кадров видеопотока: 25 fps (фиксировано)
   - Разрешение камер: 1920x1080 или 1280x720

2. **Нормативные:**
   - Соответствие 152-ФЗ "О персональных данных"
   - Соблюдение политики информационной безопасности заказчика

**Ожидания от итерации**

2. Техническая документация
3. Отчет по результатам тестирования на исторических данных
4. План пилотного внедрения с метриками эффективности

## 2. Методология (Data Scientist)

### 2.1. Постановка задачи

Задача решается комбинацией компьютерного зрения и логических правил:
1.  **Детекция объектов** — нейросеть находит на кадре ключевые элементы вагона (тормозной цилиндр, резервуар, вентиль).
2.  **Правила принятия решений** — на основе расположения найденных объектов относительно центра кадра система определяет ориентацию вагона (влево/вправо).

**Входные данные:** Кадр видеопотока (1920x1080 или 1280x720, 25 fps).
**Выходные данные:** Решение об ориентации вагона (`влево`, `вправо`).

### 2.2. Блок-схема решения

Поток обработки данных состоит из четырех ключевых этапов:
1.  **Захват и препроцессинг:** Получение видеопотока, декодирование кадров и их подготовка (изменение размера, нормализация).
2.  **ML-инференс:** Прогон кадра через нейросетевую модель для поиска референтных объектов.
3.  **Rule Engine:** Анализ геометрии найденных объектов (их расположение слева/справа в кадре) и применение логических правил для предварительной оценки ориентации.
4.  **Пост-обработка:** Агрегация результатов по нескольким кадрам одного вагона (медианная фильтрация) для вынесения финального, более стабильного решения.

### 2.3. Этапы разработки решения

#### **Этап 1: Исследовательский анализ данных (EDA)**

**Цель:** Подтвердить, что предоставленных данных достаточно для обучения качественной модели.

Мы проверим:
- **Распределение сценариев:** Достаточно ли в выборке кадров, снятых в разных условиях (ночь/день, снег/дождь/солнце)?
- **Качество разметки:** Насколько точно и согласованно размечены объекты разными экспертами.
- **Анализ признаков:** Существует ли устойчивая связь между положением объектов и ориентацией вагона.
- **Минимальный объем данных:** Расчет необходимого количества размеченных кадров (целевой минимум — 15 000) для статистической значимости результатов.

**Результат:** Отчет с выводами о достаточности данных, сложных случаях и корректировками в план разметки.

#### **Этап 2: Baseline решение**

**Цель:** Быстро получить "нижнюю границу" качества (отправную точку) для оценки эффективности основной разрабатываемой модели.

Мы используем zero-shot подход — готовую SOTA-модель (например, OwlViT), которая умеет находить объекты по текстовому описанию без предварительного обучения на наших данных. Это позволит:
- Проверить саму гипотезу: действительно ли референтные объекты различимы на кадрах.
- Получить первое значение метрик (mAP, Precision, Recall) для сравнения.

**Риски:** Ожидаемо низкое качество на сложных ракурсах и чувствительность к освещению, что подтвердит необходимость обучения кастомной модели.

#### **Этап 3: Разработка Core ML решения**

**Цель:** Создать основную модель, превосходящую baseline по качеству и скорости.

**Подход:** Обучение кастомной модели детекции на базе современной архитектуры (например, YOLOv8m). Это обеспечит лучший баланс точности и скорости инференса.

**Стратегия обучения:**
- **Оптимизатор:** AdamW — для стабильной сходимости.
- **Аугментация данных:** Применение техник, имитирующих реальные условия (изменение яркости, добавление шума, имитация дождя/снега) для повышения устойчивости модели.
- **Валидация:** Кросс-валидация по 5 блокам. Важно: разбиение будет выполняться не по отдельным кадрам, а по целым видеозаписям, чтобы избежать "подглядывания" модели в данные.

**Целевые метрики:** Основная цель — достичь точности детекции (**mAP@0.5**) не ниже **0.92** при времени инференса до **150 мс** на кадр.

#### **Этап 4: Rule Engine и принятие решений**

**Цель:** Трансформировать результаты детекции объектов в однозначный бизнес-вывод об ориентации вагона.

**Логика работы:**
1.  **Геометрический анализ:** Все обнаруженные объекты делятся на две группы по их положению в кадре (левая треть или правая треть).
2.  **Применение правил:** Существуют "жесткие" правила. Например, если тормозной цилиндр найден слева — вагон точно ориентирован влево.
3.  **Взвешенное голосование:** Если жесткие правила не сработали, система подсчитывает взвешенную сумму "голосов" от всех объектов слева и справа. Решение принимается в пользу стороны с наибольшей суммой, если разница превышает пороговое значение. В противном случае возвращается статус "не определено".

**Бизнес-валидация:** Каждую неделю мы будем демонстрировать заказчику результаты работы пайплайна на сложных случаях, сверяя их с мнением экспертов, чтобы убедиться, что логика принятия решений соответствует реальным бизнес-процессам.
