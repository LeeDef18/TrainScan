## 1. Цели и предпосылки

### 1.1. Зачем идем в разработку продукта?

**Бизнес-цель (Product Owner)**


Стратегическая цель - повышение операционной эффективности диспетчерской службы за счет автоматизации процесса идентификации ориентации подвижного состава. 


Внедрение ML-решения позволит:
- Снижение OPEX на 85% за счет автоматизации ручного труда диспетчеров
- Повышение качества мониторинга за счет снижения ошибок идентификации с 7.3% до <1%, что минимизирует риски аварий и необоснованных ремонтов.
- Масштабирование без роста затрат — подключение новых полигонов без найма персонала, обеспечение 24/7 мониторинга без сменной работы.

**Почему станет лучше, чем сейчас (Product Owner & Data Scientist)**

*Целевое состояние:*
- Автоматическая детекция референтных объектов в реальном времени
- Объективная оценка с доверительной вероятностью >95%
- Интеграция с существующей системой мониторинга через REST API
- Масштабируемость на любой полигон без дополнительных затрат

**Критерии успеха итерации (Product Owner)**
1. Достижение точности определения ориентации вагона ≥ 92% на валидационной выборке
2. Время отработки модели ≤ 350 мс на кадр

## 2. Методология (Data Scientist)

### 2.1. Постановка задачи

Задача решается в два этапа:
1. **Детекция объектов** — нейросеть YOLOv12m ищет на кадре ключевые элементы вагона(тормозной цилиндр, запасной резервуар, вентиль).
2. **Логическое правило** — простой скрипт принимает решение: если найден объект X → ориентация "лево", если найден объект Y → ориентация "право".


### 2.2. Блок-схема решения

```
Видеопоток → Разбивка на кадры → Кадр → YOLO12m (ищет объекты) 
→ Результат: найден объект / не найден → Скрипт: если объект = А → лево, если объект = Б → право
→ Сохранение результата
```

### 2.3. Этапы разработки решения

#### **Этап 1: Исследовательский анализ данных (EDA)**

**Цель:** Подтвердить, что предоставленных данных достаточно для обучения качественной модели.

Мы проверим:
- **Распределение сценариев:** достаточно ли в выборке кадров, снятых в разных условиях (ночь/день, снег/дождь/солнце)?
- **Анализ признаков:** существует ли устойчивая связь между положением объектов и ориентацией вагона.
- **Минимальный объем данных:** расчет необходимого количества размеченных кадров (целевой минимум — 15 000) для статистической значимости результатов.

**Результат:** Отчет с выводами о достаточности данных, сложных случаях и корректировками в план разметки.

---

#### **Этап 2: Быстрая проверка гипотезы**

**Цель:** Минимально жизнеспособное решение для подтверждения, что модель вообще может решать задачу.

**Подход:** Берем YOLO12m и обучаем на **маленькой размеченной выборке** (300-500 кадров), чтобы быстро проверить гипотезу.

**Что делаем:**
1. Размечаем всего 300-500 репрезентативных кадров (разные ракурсы, освещение)
2. Обучаем YOLO12m на этой малой выборке
3. Смотрим метрики: Accuracy детекции объектов

**Критерий прохождения этапа:** Модель находит объекты с точностью **>85%** на тестовой выборке.

**Результат этапа:** 
- Обученная модель (пусть и на малых данных)
- Понимание, работает ли подход в принципе
- Оценка, сколько нужно данных для целевой точности 92%

**Риски:** Минимальные — потратим несколько дней на разметку и обучение, но сэкономим недели, если гипотеза не подтвердится.

---


#### **Этап 3: Разработка Core ML решения**

**Цель:** Создать основную модель, превосходящую baseline по качеству и скорости.

**Подход:** Обучение кастомной модели детекции на базе современной архитектуры (например, YOLO12m). Это обеспечит лучший баланс точности и скорости инференса.

**Стратегия обучения:**
- **Оптимизатор:** AdamW — для стабильной сходимости.
- **Аугментация данных:** Применение техник, имитирующих реальные условия (изменение яркости, добавление шума, имитация дождя/снега) для повышения устойчивости модели.
- **Валидация:** Кросс-валидация по 5 блокам. Важно: разбиение будет выполняться не по отдельным кадрам, а по целым видеозаписям, чтобы избежать "подглядывания" модели в данные.

**Целевые метрики:** Основная цель — достичь точности детекции (**mAP@0.5**) не ниже **0.92** при времени инференса до **350 мс** на кадр.

---

#### **Этап 4: Rule Engine и принятие решений**

**Цель:** Трансформировать результаты детекции объектов в однозначный бизнес-вывод об ориентации вагона.

**Логика работы:**
1.  **Анализ объектов:** Модель обнаруживает объекты на 3-5 изображениях и сохраняет результат в JSON формате.
2.  **Применение правил:** Существуют "жесткие" правила определения ориентации каждого типа вагона. Например, если тормозной цилиндр найден на камере с правой стороны вагона типа '19-752', то ориентация - A (правильная).


**Бизнес-валидация:** Каждую неделю мы будем демонстрировать заказчику результаты работы пайплайна на сложных случаях, сверяя их с мнением экспертов, чтобы убедиться, что логика принятия решений соответствует реальным бизнес-процессам.

## 3. Подготовка пилота

### 3.1. Способ оценки пилота

**Дизайн пилота:** Теневой режим (Shadow Mode)

Система подключается параллельно к работающим диспетчерам и работает "в тени":
- Диспетчеры продолжают работать как обычно
- Наша система будет брать по 3-5 изображений из базы данных, куда срезаются кадры с вагонов для анализа(система заказчика)
- Результаты системы **не влияют** на реальную работу, только записываются в лог
- В конце дня сравниваем решения системы и диспетчера

**Почему так:**
- Безопасно — ошибки системы не нарушат реальную работу
- Позволяет накопить статистику для сравнения
- Не требует остановки текущих процессов

**Схема проведения (4 недели):**
- Неделя 1-2: Сбор данных и калибровка (проверяем, что система вообще работает)
- Неделя 3: Сбор сравнительной статистики (система vs диспетчер)
- Неделя 4: Анализ результатов и подготовка отчета

### 3.2. Что считаем успешным пилотом

**Главная метрика успеха:** Accuracy (доля совпадений решений системы с решениями диспетчера) ≥ 95%

**Дополнительные метрики:**

| Метрика | Цель | Как считаем |
|---------|------|-------------|
| Accuracy | ≥ 95% | (верные ответы системы) / (все ответы) * 100% |
| False Positive Rate | < 5% | когда система сказала "есть объект", а его нет |
| False Negative Rate | < 5% | когда система не заметила объект, но он был |
| Скорость обработки | ≤ 350 мс/кадр | время от получения кадра до выдачи ответа |

**Критерий прохождения пилота:** Достижение Accuracy ≥ 95% на валидационной выборке за последнюю неделю тестирования.

### 3.3. Подготовка пилота (вычислительные ресурсы)

**Оценка затрат для пилота:**

Для YOLO12m ориентировочные требования:
- **Для обучения:** нужна видеокарта с 8-12GB VRAM (например, NVIDIA RTX 3060/3070 или в облаке)
- **Для работы (инференс):** можно на CPU, но лучше GPU для скорости

**Что нам нужно на пилот (2-3 месяца):**

**Вариант А (локальный, бесплатно):**
- Ноутбук/ПК с видеокартой (GTX 1060 и выше)
- 16 GB RAM
- 100 GB на диске для видео
- Итог: 0 ₽ (если есть свой компьютер)

**Вариант Б (облачный, если нет своего GPU):**
- Cloud GPU (Google Colab Pro или Yandex DataSphere) — ≈ 3000-5000 ₽/мес
- Хранилище для видео — ≈ 1000 ₽/мес
- **Итого на 3 месяца:** ~15 000-18 000 ₽

**На что потратим бюджет пилота:**
1. Аренда GPU для обучения (1-2 недели активно, потом редко)
2. Хранение размеченных данных
3. Оплата сервера для работы модели в теневом режиме

**Уточнение параметров:** После первого запуска baseline модели замерим реальную скорость и точность. Если окажется, что YOLO12m слишком тяжелый (медленно работает), упростим до YOLOv11m или v8n.

## 4. Внедрение в production

### 4.1. Архитектура решения

**Ключевая особенность:** система не работает с видеопотоком в реальном времени, а получает уже готовые изображения из существующей системы учета вагонов заказчика.

**Компоненты системы:**

1. **Connector Service** — интеграционный модуль, который забирает данные из АСУ заказчика:
   - Получает информацию о новом вагоне (тип вагона, 3-5 изображений с фиксированной камеры)
   - Отправляет изображения на обработку
   - Возвращает результат обратно в систему заказчика

2. **ML Service** — содержит обученную YOLOv12m для детекции референтных объектов (вентиль, цилиндр, резервуар и т.д.)

3. **Rule Engine** — скрипт с правилами определения ориентации:
   - Для каждого типа вагона (например, '19-752') задано правило: "если на изображении с камеры X обнаружен объект Y → ориентация правильная / неправильная"
   - Правила хранятся отдельно и могут изменяться без переобучения модели

4. **Storage** — база данных для хранения результатов и журналирования

**Схема взаимодействия:**
```
АСУ заказчика (учет вагонов) 
    ↓ (новый вагон + 3-5 изображений)
Connector Service 
    ↓
ML Service (детекция объектов) 
    ↓ (список найденных объектов)
Rule Engine (применение правил для типа вагона)
    ↓ (решение: ориентация правильная/нет)
Connector Service 
    ↓
АСУ заказчика (получение результата)
    ↓
Storage (логирование для анализа)
```

### 4.2. Инфраструктура и масштабирование

**Выбор:** модуль, развернутый на сервере заказчика (on-premise) или в контейнере Docker.

**Почему так:**
- Данные (изображения вагонов) не нужно передавать наружу — безопаснее для заказчика
- Интеграция происходит на стороне заказчика, мы только поставляем свой модуль
- Не требуется сложной оркестрации — сервис запускается по событию от АСУ

**Плюсы:**
- Простота внедрения — один модуль, который подключается к существующей системе
- Безопасность данных — изображения не покидают контур заказчика
- Минимальные требования к инфраструктуре
- Легко тестировать и обновлять

**Минусы:**
- Зависимость от стабильности работы АСУ заказчика
- Сложнее мониторить производительность

### 4.3. Требования к работе системы (SLA)

| Параметр | Целевое значение | Пояснение |
|----------|------------------|-----------|
| Доступность | 97% | Система должна быть доступна в рабочее время |
| Пропускная способность | 100-200 изображений/сек | Типичная нагрузка для одного полигона |
| Время восстановления | ≤ 2 часа | При сбоях |

### 4.4. Безопасность системы

**Потенциальные уязвимости:**

1. **Несанкционированный доступ к модулю**
   → Защита: модуль работает внутри контура заказчика, доступ только через АСУ

2. **Подмена модели или правил**
   → Защита: контроль целостности файлов, версионирование модели и правил

3. **Отказ в обслуживании**
   → Защита: очередь запросов, таймауты, повторные попытки при сбоях

### 4.5. Безопасность данных

**Что у нас за данные:**
- Изображения вагонов (технические объекты, не люди)
- Техническая информация о типах вагонов
- Результаты детекции и принятые решения

**Соответствие требованиям:**
- Изображения не содержат персональных данных (камера направлена на вагоны)
- Все данные остаются в инфраструктуре заказчика
- Мы не храним данные у себя, только логируем результаты обработки

**Вывод:** Нарушений 152-ФЗ и GDPR нет, так как работаем с техническими объектами в контуре заказчика.

### 4.7. Интеграционные точки

**Способ интеграции:** АСУ заказчика вызывает наш модуль по API (HTTP/REST) при появлении нового вагона.

**Входные данные:**
- Идентификатор вагона
- Тип вагона (например, '19-752')
- Ссылки на 3-5 изображений или сами изображения (base64)
- Идентификатор камеры (всегда одна и та же)

**Выходные данные:**
- Решение об ориентации: "правильная" / "неправильная" / "не удалось определить"
- Уверенность решения (confidence score)
- Список обнаруженных объектов (для журналирования)

**Процесс интеграции:**
1. Заказчик предоставляет тестовый доступ к АСУ
2. Мы разворачиваем свой модуль и настраиваем вызовы
3. Проводим интеграционное тестирование (10-20 вагонов)
4. Передаем модуль в эксплуатацию

### 4.8. Риски

| Риск | Вероятность | Что делаем |
|------|-------------|------------|
| Правила для типов вагонов неполные | Средняя | Собираем информацию от заказчика, пополняем базу правил |
| Модель не обнаруживает объекты на новых типах вагонов | Средняя | Плановое дообучение модели раз в квартал |
| Сбои в АСУ заказчика | Средняя | Логируем ошибки, предусматриваем повторные попытки |
| Изменение ракурса камеры | Низкая | Калибровка при изменении, быстрое дообучение |
| Заказчик хочет добавлять новые типы вагонов | Низкая | Правила хранятся отдельно — добавляем без переобучения модели |
