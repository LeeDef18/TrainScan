# ML System Design Doc - [RU]
## Дизайн ML системы - \<Автоматизаци определения ориентации железнодорожных вагонов\> \<MVP 1\>

## 1. Цели и предпосылки

### 1.1. Зачем идем в разработку продукта?

**Бизнес-цель (Product Owner)**


Стратегическая цель - повышение операционной эффективности диспетчерской службы за счет автоматизации процесса идентификации ориентации подвижного состава. 


Внедрение ML-решения позволит:
- Снижение OPEX на 85% за счет автоматизации ручного труда диспетчеров
- Повышение качества мониторинга за счет снижения ошибок идентификации с 7.3% до <1%, что минимизирует риски аварий и необоснованных ремонтов.
- Масштабирование без роста затрат — подключение новых полигонов без найма персонала, обеспечение 24/7 мониторинга без сменной работы.

**Почему станет лучше, чем сейчас (Product Owner & Data Scientist)**


*Текущее состояние:*
- Ручной анализ видеозаписей операторами
- Субъективность оценки при плохой видимости
- Отсутствие единой базы данных ориентации вагонов
- Невозможность масштабирования без увеличения штата

*Целевое состояние:*
- Автоматическая детекция референтных объектов в реальном времени
- Объективная оценка с доверительной вероятностью >95%
- Интеграция с существующей системой мониторинга через REST API
- Масштабируемость на любой полигон без дополнительных затрат

**Критерии успеха итерации (Product Owner)**
1. Достижение точности определения ориентации вагона ≥ 92% на валидационной выборке
2. Время отработки модели ≤ 350 мс на кадр


### 1.2. Бизнес-требования и ограничения

**Бизнес-требования**
- Система должна определять ориентацию вагона с точностью ≥90%

2. **Нормативные:**
   - Соответствие 152-ФЗ "О персональных данных"
   - Соблюдение политики информационной безопасности заказчика

## 2. Методология (Data Scientist)

### 2.1. Постановка задачи

Задача решается в два этапа:
1. **Детекция объектов** — нейросеть YOLOv12m ищет на кадре ключевые элементы вагона(тормозной цилиндр, запасной резервуар, цилиндр).
2. **Логическое правило** — простой скрипт принимает решение: если найден объект X → ориентация "лево", если найден объект Y → ориентация "право".


### 2.2. Блок-схема решения

```
Видеопоток → Разбивка на кадры → Кадр → YOLOv12m (ищет объекты) 
→ Результат: найден объект / не найден → Скрипт: если объект = А → лево, если объект = Б → право
→ Сохранение результата
```

### 2.3. Этапы разработки решения

#### **Этап 1: Исследовательский анализ данных (EDA)**

**Цель:** Подтвердить, что предоставленных данных достаточно для обучения качественной модели.

Мы проверим:
- **Распределение сценариев:** достаточно ли в выборке кадров, снятых в разных условиях (ночь/день, снег/дождь/солнце)?
- **Анализ признаков:** существует ли устойчивая связь между положением объектов и ориентацией вагона.
- **Минимальный объем данных:** расчет необходимого количества размеченных кадров (целевой минимум — 15 000) для статистической значимости результатов.

**Результат:** Отчет с выводами о достаточности данных, сложных случаях и корректировками в план разметки.

Вот обновленный этап 2 с фокусом на быструю проверку гипотезы через обучение на малой выборке:

---

#### **Этап 2: Быстрая проверка гипотезы**

**Цель:** Минимально жизнеспособное решение для подтверждения, что модель вообще может решать задачу.

**Подход:** Берем YOLOv12m и обучаем на **маленькой размеченной выборке** (300-500 кадров), чтобы быстро проверить гипотезу.

**Что делаем:**
1. Размечаем всего 300-500 репрезентативных кадров (разные ракурсы, освещение)
2. Обучаем YOLOv12m на этой малой выборке
3. Смотрим метрики: Accuracy детекции объектов

**Критерий прохождения этапа:** Модель находит объекты с точностью **>85%** на тестовой выборке.

**Результат этапа:** 
- Обученная модель (пусть и на малых данных)
- Понимание, работает ли подход в принципе
- Оценка, сколько нужно данных для целевой точности 92%

**Риски:** Минимальные — потратим несколько дней на разметку и обучение, но сэкономим недели, если гипотеза не подтвердится.

---


#### **Этап 3: Разработка Core ML решения**

**Цель:** Создать основную модель, превосходящую baseline по качеству и скорости.

**Подход:** Обучение кастомной модели детекции на базе современной архитектуры (например, YOLO12m). Это обеспечит лучший баланс точности и скорости инференса.

**Стратегия обучения:**
- **Оптимизатор:** AdamW — для стабильной сходимости.
- **Аугментация данных:** Применение техник, имитирующих реальные условия (изменение яркости, добавление шума, имитация дождя/снега) для повышения устойчивости модели.
- **Валидация:** Кросс-валидация по 5 блокам. Важно: разбиение будет выполняться не по отдельным кадрам, а по целым видеозаписям, чтобы избежать "подглядывания" модели в данные.

**Целевые метрики:** Основная цель — достичь точности детекции (**mAP@0.5**) не ниже **0.92** при времени инференса до **150 мс** на кадр.

#### **Этап 4: Rule Engine и принятие решений**

**Цель:** Трансформировать результаты детекции объектов в однозначный бизнес-вывод об ориентации вагона.

**Логика работы:**
1.  **Геометрический анализ:** Все обнаруженные объекты делятся на две группы по их положению в кадре (левая треть или правая треть).
2.  **Применение правил:** Существуют "жесткие" правила. Например, если тормозной цилиндр найден слева — вагон точно ориентирован влево.
3.  **Взвешенное голосование:** Если жесткие правила не сработали, система подсчитывает взвешенную сумму "голосов" от всех объектов слева и справа. Решение принимается в пользу стороны с наибольшей суммой, если разница превышает пороговое значение. В противном случае возвращается статус "не определено".

**Бизнес-валидация:** Каждую неделю мы будем демонстрировать заказчику результаты работы пайплайна на сложных случаях, сверяя их с мнением экспертов, чтобы убедиться, что логика принятия решений соответствует реальным бизнес-процессам.

## 4. Внедрение в production (если потребуется)

*Для учебного проекта production-внедрение не обязательно, но опишем, как это могло бы работать, если пилот пройдет успешно.*

### 4.1. Архитектура решения

**Компоненты системы:**

1. **Video Ingestion Service** — забирает видеопоток с камер, режет на кадры
2. **ML Service** — содержит обученную YOLOv12m, получает кадр → возвращает найденные объекты
3. **Logic Service** — скрипт с правилами "если объект = А → лево, если = Б → право"
4. **API Gateway** — единая точка входа для внешних запросов
5. **Storage** — база данных для хранения результатов

**Схема взаимодействия:**
```
Камера → Video Service → [кадр] → ML Service → [объекты] → Logic Service → [ориентация] → Storage
                                                                              →
                                                                              → API (по запросу)
```

**Методы API (простые):**
- `POST /predict` — отправить кадр, получить ориентацию
- `GET /results?from=date&to=date` — получить статистику за период
- `GET /health` — проверить, что система жива

### 4.2. Инфраструктура и масштабирование

**Выбор:** Docker + один сервер (потом можно будет расширить)

**Почему так:**
- Для пилота и небольшой нагрузки достаточно одного сервера
- Docker позволяет упаковать всё с зависимостями и не мучиться с установкой
- Если нагрузка вырастет, Docker Compose легко превратить в Kubernetes

**Плюсы:**
✅ Простота настройки
✅ Минимальные затраты
✅ Всё в одном месте, легко отлаживать

**Минусы:**
❌ Если сервер упадет, система встанет
❌ Сложно масштабировать под большой поток камер

**Для учебного проекта — идеально.**

### 4.3. Требования к работе системы (SLA)

| Параметр | Целевое значение | Пояснение |
|----------|------------------|-----------|
| Доступность | 95% (в рабочее время) | Может иногда падать, но быстро поднимаем |
| Время ответа | ≤ 500 мс | От получения кадра до ответа |
| Пропускная способность | 10 кадров/сек | Хватит для 1-2 камер |
| Время восстановления | ≤ 1 час | Если упало, чиним в течение часа |

*Для учебного проекта жесткие 99.9% не нужны.*

### 4.4. Безопасность системы

**Потенциальные уязвимости:**

1. **Подмена модели** — кто-то может подложить свою модель вместо нашей
   → Защита: храним модель в защищенном хранилище, проверяем хеш-сумму

2. **Перегрузка API** — кто-то начнет слать тысячи запросов
   → Защита: ограничение числа запросов с одного IP (rate limiting)

3. **Доступ без авторизации** — любой может вызывать наш API
   → Защита: простой API-ключ в заголовке запроса

### 4.5. Безопасность данных (GDPR и 152-ФЗ)

**Что у нас за данные:**
- Видео с железнодорожных камер (вагоны, техника)
- Людей в кадре **не должно быть** (камеры направлены на пути)

**Риски:**
- Если в кадр попал человек — это уже персональные данные
- Видео могут быть чувствительными для заказчика

**Что делаем:**
1. Заключаем соглашение о неразглашении (NDA)
2. Храним данные в зашифрованном виде
3. Доступ к данным только у участников проекта
4. Если используем облако — выбираем российское (Яндекс.Облако) для соблюдения 152-ФЗ
5. После окончания проекта данные удаляем

**Вывод:** Если в кадры не попадают люди, то нарушения GDPR/152-ФЗ нет.

### 4.6. Издержки (расчет на месяц для production)

*Если бы система работала постоянно на одном полигоне*

| Статья расходов | Стоимость/мес | Примечание |
|-----------------|---------------|------------|
| Сервер (облачный) | 5 000-7 000 ₽ | 4 vCPU, 8 GB RAM, небольшой GPU или CPU |
| Хранение данных | 2 000 ₽ | 500 GB (видео храним 2 недели) |
| Сеть (трафик) | 1 000 ₽ | Запросы к API |
| Поддержка (человек) | 20 000-30 000 ₽ | 0.25 ставки (не полный день) |
| **ИТОГО** | **28 000-40 000 ₽/мес** | |

**Как сэкономить:**
- Крутить на своем железе (0 ₽ за сервер)
- Хранить только результаты, не хранить видео
- Отказаться от поддержки (сами чиним)

### 4.7. Интеграционные точки (API)

**1. Внутренние (между сервисами):**
- Video Service → ML Service: передача кадра (HTTP или gRPC)
- ML Service → Logic Service: передача списка объектов (внутренний вызов)

**2. Внешние (для заказчика):**

```yaml
POST /api/v1/orientation
Headers:
  X-API-Key: ваш_ключ
Body: изображение (JPEG/PNG)
Response:
{
  "orientation": "left/right/unknown",
  "confidence": 0.95,
  "object_found": "brake_cylinder",
  "processing_time_ms": 150
}

GET /api/v1/health
Response:
{
  "status": "ok",
  "model": "yolov12m",
  "uptime": "5d 3h 12m"
}
```

### 4.8. Риски

| Риск | Вероятность | Что делаем |
|------|-------------|------------|
| Модель плохо работает на новых данных | Средняя | Собираем больше данных, делаем валидацию |
| Не хватило размеченных данных | Средняя | Дополнительная разметка (заложить бюджет) |
| Медленная работа YOLOv12m | Низкая | Перейдем на YOLOv8n (легче и быстрее) |
| Заказчик не доволен точностью | Средняя | Четко фиксируем метрики успеха до пилота |
| Видеопоток нестабильный | Низкая | Добавляем повторные попытки, буферизацию |
| Закончились деньги на облако | Средняя | Переезжаем на локальный компьютер |
