# ML System Design Doc - [RU]
## Дизайн ML системы - \<Автоматизаци определения ориентации железнодорожных вагонов\> \<MVP 1\>

## 1. Цели и предпосылки

### 1.1. Зачем идем в разработку продукта?

**Бизнес-цель (Product Owner)**


Стратегическая цель - повышение операционной эффективности диспетчерской службы за счет автоматизации процесса идентификации ориентации подвижного состава.


Внедрение ML-решения позволит:
- Снижение OPEX на 85% за счет автоматизации ручного труда диспетчеров
- Повышение качества мониторинга за счет снижения ошибок идентификации, что минимизирует риски аварий.
- Масштабирование без роста затрат — подключение новых полигонов без найма персонала, обеспечение 24/7 мониторинга без сменной работы.

**Почему станет лучше, чем сейчас (Product Owner & Data Scientist)**


*Текущее состояние:*
- Ручной анализ видеозаписей операторами
- Субъективность оценки при плохой видимости
- Невозможность масштабирования без увеличения штата

*Целевое состояние:*
- Автоматическая детекция референтных объектов
- Объективная оценка с доверительной вероятностью >95%
- Интеграция с существующей системой мониторинга
- Масштабируемость на любой полигон без дополнительных затрат

**Критерии успеха итерации (Product Owner)**
1. Достижение точности определения ориентации вагона ≥ 92% на валидационной выборке
2. Время отработки модели ≤ 350 мс на кадр


### 1.2. Бизнес-требования и ограничения

**Бизнес-требования**
- Система должна определять ориентацию вагона с точностью ≥90%
- Время обработки состава ≤ 5 минут
- Работа в условиях ограниченной видимости (ночь, осадки)
- Интеграция с АСУ "Транс-Телематика"


**Бизнес-ограничения**
1. **Технические:**
   - Частота кадров видеопотока: 25 fps (фиксировано)
   - Разрешение камер: 1280x720

2. **Нормативные:**
   - Соответствие 152-ФЗ "О персональных данных"
   - Соблюдение политики информационной безопасности заказчика

**Ожидания от итерации**
1. Рабочий прототип ML-модели
2. Техническая документация
3. Отчет по результатам тестирования на валидационных данных
4. План пилотного внедрения с метриками эффективности

## 2. Методология (Data Scientist)

### 2.1. Постановка задачи

Задача решается в два этапа:
1. **Детекция объектов** — нейросеть YOLO ищет на кадре ключевые элементы вагона(тормозной цилиндр, запасной резервуар, вентиль).
2. **Логическое правило** — скрипт принимает решение: когда модель распознает объект/объекты, тогда скрипт делает вывод на основе базы правил ориентации для каждого типа вагона - правильная (A)/неправильная (B) ориентация. К примеру, если модель на камере с правой стороны вагона типа '19-752' обнаружила вентиль → скрипт сверяет в таблице правило ориентации и сохраняет результат.


### 2.2. Блок-схема решения

```
АСУ → кадры вагона → YOLO (ищет объекты)
→ Результат: найден объект / не найден → Скрипт: сверка с таблицей ориентации
→ Сохранение результата
```

### 2.3. Этапы разработки решения

#### **Этап 1: Исследовательский анализ данных (EDA)**

**Цель:** подтвердить, что предоставленных данных достаточно для обучения качественной модели.


Мы проверим:
- **Распределение сценариев:** достаточно ли в выборке кадров, снятых в разных условиях (ночь/день, снег/дождь/солнце)?
- **Анализ признаков:** существует ли устойчивая связь между положением объектов и ориентацией вагона.
- **Минимальный объем данных:** расчет необходимого количества размеченных кадров для статистической значимости результатов.

**Результат:** датасет состоит из 100 000+ изображений(кадров) разных типов вагона и ракурсов с камер. Данные разбиты по категориям - технические и погодные дефекты (затемнения/блики, дождь/снег, грязь, наклон камеры). Всего 4 камеры - с каждой стороны по 2 камеры (снизу, сверху). Разные ракурсы помогают получить полную картину логики распределения объектов для каждого типа вагона. Разрешение изображений 1280x720, формат jpg, jpeg, png, фильтры отсутствуют. Целевой минимум данных — 10 000 для статистической значимости результатов.

---

#### **Этап 2: Подготовка данных**

**Цель:** подготовить данные для обучения.

- **Формирование выборки:** берем 15000 случайных изображений с одной камеры и разделяем по принципу 70% на обучающую, 15% на валидационную и 15% на тестовую выборки.
- **Разметка данных:** размечаем 85% изображений (12750) при помощи CVAT и конвертируем в YOLO-формат.

---

#### **Этап 3: Быстрая проверка гипотезы**

**Цель:** минимально жизнеспособное решение для подтверждения, что модель вообще может решать задачу.

**Подход:** берем YOLO11n и обучаем на **маленькой размеченной выборке** (1000-1500 кадров), чтобы быстро проверить гипотезу.

**Что делаем:**
1. Берем всего 1000-1500 репрезентативных кадров (разные ракурсы, освещение)
2. Обучаем YOLO11n на этой малой выборке
3. Смотрим метрики: Accuracy детекции объектов

**Критерий прохождения этапа:** модель находит объекты с точностью **>85%** на тестовой выборке.

**Результат этапа:**
- Обученная модель (пусть и на малых данных)
- Понимание, работает ли подход в принципе
- Оценка, сколько нужно данных для целевой точности > 92%

**Риски:** минимальные — потратим несколько дней на разметку и обучение, но сэкономим недели, если гипотеза не подтвердится.

---

#### **Этап 4: Разработка Core ML решения**

**Цель:** обучить основную модель на полном объеме данных для достижения целевых метрик.

**Подход:** обучение кастомной модели детекции YOLOv12m на размеченном датасете из 15 000 кадров.

**Процесс обучения:**
- Обучаем YOLOv12m на тренировочной выборке
- После каждой эпохи проверяем качество на валидационной выборке
- Выбираем лучшую версию модели (которая показала максимальный результат на валидации)
- Финальную модель тестируем один раз на тестовой выборке (эта оценка идет в отчет)

**Аугментация данных:**
Применяем техники, имитирующие реальные условия (изменение яркости, добавление шума, имитация дождя/снега) для повышения устойчивости модели. Аугментации применяются только к тренировочной выборке.

**Целевые метрики:**
| Метрика | Цель | Где проверяем |
|---------|------|----------------|
| mAP@0.5 | ≥ 0.95 | На тестовой выборке |
| Precision | ≥ 0.95 | На тестовой выборке |
| Recall | ≥ 0.90 | На тестовой выборке |
| Время инференса | ≤ 350 мс на кадр | На тестовом окружении |

---

#### **Этап 5: Rule Engine и принятие решений**

**Цель:** трансформировать результаты детекции объектов в однозначный бизнес-вывод об ориентации вагона.

**Логика работы:**
1.  **Анализ объектов:** модель обнаруживает объекты на 3-5 изображениях и сохраняет взвешанный результат в JSON формате.
2.  **Применение правил:** существуют "жесткие" правила определения ориентации каждого типа вагона. Например, если тормозной цилиндр найден на камере с правой стороны вагона типа '19-752', то ориентация - A (правильная).


**Бизнес-валидация:** каждую неделю мы будем демонстрировать заказчику результаты работы пайплайна на сложных случаях, сверяя их с мнением экспертов, чтобы убедиться, что логика принятия решений соответствует реальным бизнес-процессам.

## 3. Подготовка пилота

### 3.1. Способ оценки пилота

**Дизайн пилота:** теневой режим (Shadow Mode)

Система подключается параллельно к работающим диспетчерам и работает "в тени":
- Диспетчеры продолжают работать как обычно
- Наша система будет брать по 3-5 изображений из базы данных, куда срезаются кадры с вагонов для анализа(система заказчика)
- Результаты системы **не влияют** на реальную работу, только записываются в лог
- В конце дня сравниваем решения системы и итоговые отчеты по ориентации

**Почему так:**
- Безопасно — ошибки системы не нарушат реальную работу
- Позволяет накопить статистику для сравнения
- Не требует остановки текущих процессов

**Схема проведения (4 недели):**
- Неделя 1-2: Сбор данных и калибровка (проверяем, что система вообще работает; исправление багов; )
- Неделя 2-3: Сбор сравнительной статистики (система vs отчеты; за неделю пройдет 700-1200 вагонов для статистически значимых результатов)
- Неделя 3-4: Анализ результатов и подготовка отчета

---

### 3.2. Что считаем успешным пилотом

**Главная метрика успеха:** Accuracy (доля совпадений решений системы с итоговыми отчетами) ≥ 95%

**Дополнительные метрики:**

| Метрика | Цель | Как считаем |
|---------|------|-------------|
| Accuracy | ≥ 95% | (верные ответы системы) / (все ответы) * 100% |
| False Positive Rate | < 5% | когда система сказала "есть объект", а его нет |
| False Negative Rate | < 5% | когда система не заметила объект, но он был |
| Скорость обработки | ≤ 350 мс/кадр | время от получения кадра до выдачи ответа |

**Критерий прохождения пилота:** Достижение Accuracy ≥ 95% на валидационной выборке за последнюю неделю тестирования.

---

### 3.3. Подготовка пилота (вычислительные ресурсы)

**Оценка затрат для пилота:**

Для YOLO12m ориентировочные требования:
- **Для обучения:** нужна видеокарта с 8-12GB VRAM (например, NVIDIA RTX 3060/3070 или в облаке)
- **Для работы (инференс):** можно на CPU, но лучше GPU для скорости

**Что нам нужно на пилот (2-3 месяца):**

**Вариант А (локальный, бесплатно):**
- Ноутбук/ПК с видеокартой (GTX 1060 и выше)
- 16 GB RAM
- 100 GB на диске для видео
- Итог: 0 ₽ (если есть свой компьютер)

**Вариант Б (облачный, если нет своего GPU):**
- Cloud GPU (Google Colab Pro или Yandex DataSphere) — ≈ 3000-5000 ₽/мес
- Хранилище для видео — ≈ 1000 ₽/мес
- **Итого на 3 месяца:** ~15 000-18 000 ₽

**На что потратим бюджет пилота:**
1. Аренда GPU для обучения (1-2 недели активно, потом редко)
2. Хранение размеченных данных
3. Оплата сервера для работы модели в теневом режиме

**Уточнение параметров:** После первого запуска baseline модели замерим реальную скорость и точность. Если окажется, что YOLO12m слишком тяжелый (медленно работает), упростим до YOLOv11m или v8n.

## 4. Внедрение в production

### 4.1. Целевой production-контур

Для production целевая схема должна быть не просто REST API с моделью, а устойчивый on-premise сервис обработки событий, встроенный в контур заказчика и рассчитанный на длительную эксплуатацию.

**Базовые принципы production-решения:**

* Развертывание внутри инфраструктуры заказчика без вывода изображений за пределы защищенного контура.
* Разделение online-контура инференса и offline-контура обучения/переобучения.
* Контейнеризация всех сервисов и воспроизводимое развертывание через CI/CD.
* Идемпотентная обработка задач, чтобы повторная отправка одного и того же вагона не приводила к дублированию результата.
* Наблюдаемость на уровне сервиса, модели и бизнес-метрик.

**Целевая схема:**

```text
АСУ / источник кадров / БД заказчика
        ↓
Ingestion Adapter
        ↓
Очередь задач / job table
        ↓
Inference Worker(s) with YOLO
        ↓
Rule Engine
        ↓
Result Writer
        ↓
БД результатов / интеграционная таблица / API заказчика
        ↓
Monitoring / Logs / Alerts / Audit
```

**Почему именно такая схема:**

* Она снимает жесткую связность между поступлением кадров и инференсом.
* Позволяет масштабировать обработку горизонтально через несколько worker-ов.
* Упрощает ретраи, контроль очереди и диагностику инцидентов.
* Дает возможность эволюционно перейти от MVP API к промышленному контуру без полной переработки доменной логики.

---

### 4.2. Архитектура сервисов

**Состав production-решения:**

| Компонент | Назначение |
|-----------|------------|
| API / Admin Service | healthcheck, ручной запуск диагностики, служебные эндпоинты, версионирование модели |
| Ingestion Adapter | получает информацию о новых кадрах или читает staging-таблицу заказчика |
| Queue / Job Store | буферизует задачи обработки и хранит статус выполнения |
| Inference Worker | загружает модель, выполняет батчевый или поштучный инференс |
| Rule Engine | вычисляет итоговую ориентацию по результатам детекции и таблице правил |
| Result Writer | записывает финальный результат, признаки качества и технический статус |
| Model Registry / Artifact Storage | хранит версии модели, rule table и служебные артефакты |
| Monitoring Stack | метрики, централизованные логи, трассировка, алерты |

**Логика обработки одной сущности:**

1. Система заказчика сохраняет 3-5 кадров и метаданные вагона.
2. `Ingestion Adapter` создает задачу обработки с уникальным `wagon_id`/`inspection_id`.
3. `Inference Worker` забирает задачу из очереди, валидирует входные данные и выполняет инференс.
4. `Rule Engine` агрегирует детекции по нескольким кадрам и определяет итоговую ориентацию `A/B`.
5. `Result Writer` сохраняет результат, confidence, список подтвержденных объектов, версию модели и версию rule table.
6. Система мониторинга фиксирует технические и бизнес-метрики обработки.

**Что остается от текущего MVP:**

* Текущий FastAPI-сервис может быть сохранен как synchronous entrypoint для ручной проверки, отладки и тестовых прогонов.
* Бизнес-логика `PredictUseCase`, rule engine и batch-агрегация переиспользуются внутри worker-процесса.
* Docker и reverse proxy уже соответствуют минимальной базе для первого production-like контура.

---

### 4.3. Инфраструктура и масштабируемость

**Целевой способ развертывания:**

* Основной вариант: `on-premise` на сервере или виртуальной машине заказчика.
* Формат поставки: Docker Compose на раннем этапе, далее Kubernetes/OpenShift при росте требований к отказоустойчивости.
* Вычисления: GPU-пул для инференса как основной режим, CPU только как fallback для деградационного сценария.

**Минимальный production-контур:**

* `1 x nginx/reverse proxy`
* `1 x API/admin service`
* `1-2 x inference worker`
* `1 x PostgreSQL/внутренняя БД статусов` или интеграционная job table в БД заказчика
* `1 x Redis/RabbitMQ` при использовании внешней очереди
* стек мониторинга: Prometheus + Grafana + централизованный сбор логов

**Масштабирование:**

* Горизонтальное масштабирование через добавление `Inference Worker`.
* Вертикальное масштабирование через выделение GPU с большим VRAM для более тяжелой модели.
* Отдельное масштабирование ingestion и inference-контуров, чтобы всплеск входящего потока не приводил к отказу API.

**Проектный ориентир по росту:**

1. Этап 1: один сервер, один worker, обработка в near-real-time.
2. Этап 2: несколько worker-ов и очередь задач, если поток вагонов начинает расти.
3. Этап 3: active-standby или кластерный deployment с резервированием критичных компонентов.

---

### 4.4. Надежность и эксплуатация

**Целевые SLO на production-этапе:**

| Показатель | Целевое значение |
|-----------|------------------|
| Доступность сервиса инференса | ≥ 99.5% в месяц |
| P95 времени обработки одного кадра | ≤ 350 мс |
| P95 времени обработки одного вагона | ≤ 30 секунд при наличии всех кадров |
| Уровень успешного завершения задач | ≥ 99% |

**Требования к устойчивости:**

* Перезапуск сервисов должен происходить автоматически.
* Все задачи должны иметь статусы `new`, `processing`, `done`, `failed`, `retry`.
* Повторная обработка после сбоя не должна приводить к потере данных.
* Если модель или rule table недоступны, система должна переходить в controlled failure с понятным техническим статусом, а не записывать некорректный бизнес-результат.

**Обязательные operational-механизмы:**

* `healthcheck`, `readiness`, `liveness` для каждого сервиса.
* таймауты на чтение изображений, инференс и запись результата;
* ограничение размера входного изображения и количества кадров в одной задаче;
* retry policy с backoff только для временных ошибок;
* dead-letter механизм для задач, которые не удалось обработать автоматически.

---

### 4.5. Observability и ML-мониторинг

В production для ML-системы недостаточно только application logs. Нужно отдельно наблюдать техническое состояние сервиса и качество поведения модели на реальном потоке.

**Что мониторим технически:**

* количество входящих задач в минуту;
* длину очереди и время ожидания в очереди;
* latency по стадиям: загрузка кадра, preprocessing, inference, rule evaluation, запись результата;
* долю ошибок по типам: invalid image, timeout, model load error, DB write error;
* использование CPU, RAM, GPU, VRAM, диска и сети.

**Что мониторим как ML-систему:**

* распределение confidence по классам;
* долю кадров без детекций;
* частоту выдачи ориентаций `A/B` по типам вагонов;
* долю uncertain/technical-failure результатов;
* расхождение с ручной разметкой или отчетами заказчика на контрольной подвыборке;
* drift по качеству входных кадров и по распределению признаков.

**Алерты должны срабатывать при:**

* резком росте очереди;
* падении успешности обработки;
* деградации latency;
* росте доли пустых детекций;
* аномальном изменении распределения результатов по конкретному типу вагона.

---

### 4.6. Безопасность системы и данных

**Требования безопасности:**

* сервис работает только во внутреннем контуре заказчика;
* доступ к БД, очередям и object storage ограничивается service account-ами;
* у каждого компонента минимально необходимые права;
* все действия, влияющие на результат, должны быть аудируемыми.

**Минимальный набор мер:**

* TLS на внутренних и внешних точках подключения, если это допускается контуром заказчика;
* хранение секретов в защищенном vault/secret store, а не в коде или образе;
* allowlist на сетевые соединения между сервисами;
* проверка входных файлов: формат, размер, декодируемость, защита от malformed image payload;
* журналирование: кто, когда, какой результат записал и на основании какой версии модели;
* регулярное сканирование контейнерных образов на уязвимости.

**Соответствие требованиям:**

* данные не покидают инфраструктуру заказчика;
* при появлении персональных данных в смежных системах потребуется отдельная проверка соответствия 152-ФЗ и внутренним политикам ИБ;
* production-контур должен проходить внутреннее согласование с ИБ-службой заказчика до ввода в промышленную эксплуатацию.

---

### 4.7. Управление версиями модели и релизами

Для production важно версионировать не только код, но и ML-артефакты.

**Что должно быть версионировано:**

* контейнерный образ приложения;
* файл модели;
* таблица правил ориентации;
* конфигурация inference thresholds;
* схема выходного результата.

**Требования к релизному процессу:**

1. Новая модель проходит offline-валидацию на тестовом наборе.
2. Затем раскатывается в shadow/canary-режиме на части потока.
3. Только после подтверждения метрик становится основной версией.
4. Для каждой записи результата сохраняются `model_version` и `rules_version`.
5. Должна существовать быстрая процедура rollback на предыдущую стабильную версию.

**Практический смысл:**

Без этой дисциплины невозможно разбирать инциденты, сравнивать версии модели и доказывать, почему система выдала конкретный результат в конкретную дату.

---

### 4.8. Интеграционные точки

Целевой production не должен быть завязан только на один вариант интеграции. Желательно предусмотреть два совместимых сценария.

**Сценарий 1. Интеграция через БД заказчика**

* чтение кадров и метаданных из staging-таблицы или представления;
* запись результата в отдельную таблицу результатов;
* использование технических статусов обработки и timestamps.

**Сценарий 2. Интеграция через внутренний API или очередь событий**

* заказчик публикует событие о готовности набора кадров;
* наша система забирает payload, обрабатывает его и отправляет результат обратно;
* такой вариант предпочтительнее при дальнейшем масштабировании и снижает нагрузку на БД АСУ.

**Рекомендуемая структура записи результата:**

```json
{
  "wagon_id": "12345",
  "inspection_id": "2026-03-26-0001",
  "frames_used": 5,
  "detections": [
    {"class": "brake_valve", "confidence": 0.97}
  ],
  "orientation": "A",
  "decision_status": "done",
  "decision_reason": "rule_matched",
  "model_version": "yolo12m_2026_03_20",
  "rules_version": "rules_v5",
  "processed_at": "2026-03-26T12:00:00Z"
}
```

**Дополнительно в интеграции стоит предусмотреть:**

* уникальный ключ идемпотентности;
* технический код ошибки для неуспешной обработки;
* поле признака ручной верификации результата;
* SLA на доступность входного источника данных со стороны заказчика.

---

### 4.9. Издержки и эксплуатационная модель

В production основная стоимость возникает не только из-за сервера, но и из-за сопровождения ML-системы как продукта.

**Основные статьи затрат:**

| Статья | Комментарий |
|--------|-------------|
| Серверные ресурсы | CPU/GPU, RAM, диск, резервирование |
| Системное ПО | мониторинг, логирование, backup, secret management |
| Эксплуатация | обновления, инциденты, дежурство, контроль SLA |
| ML-сопровождение | переоценка качества, доразметка, дообучение, релизы моделей |

**Вывод по cost model:**

* Если инфраструктура заказчика уже существует, CAPEX на старт минимален.
* OPEX в production будет определяться не хостингом, а регулярной поддержкой сервиса и модели.
* В бюджет нужно закладывать отдельный ресурс на MLOps/поддержку качества, иначе система быстро деградирует после ввода в эксплуатацию.

---

### 4.10. Основные риски production-внедрения

| Риск | Описание | Митигирование |
|------|----------|---------------|
| Перегрузка БД АСУ | частое чтение кадров влияет на контур заказчика | staging-таблицы, event-driven интеграция, индексы, rate limit |
| Деградация качества входных кадров | новые камеры, загрязнение, смещение ракурса | мониторинг качества данных, drift detection, регламент проверки камер |
| Медленный инференс | модель не укладывается в SLA на CPU | GPU-профиль, оптимизация модели, ONNX/TensorRT на следующем этапе |
| Ошибки в таблице правил | модель детектирует корректно, но бизнес-вывод неверный | versioning rule table, тест-кейсы на правила, экспертная валидация |
| Невоспроизводимость результата | невозможно понять, какая версия модели приняла решение | обязательное хранение model_version/rules_version |
| Рост очереди при пиковых нагрузках | сервис не успевает обработать поток | autoscaling worker-ов, backpressure, приоритизация задач |
| Тихая деградация ML-модели | система формально работает, но качество падает | контрольная выборка, shadow evaluation, регулярный quality review |
| Отсутствие rollback-процесса | неудачный релиз модели останавливает работу | blue-green/canary rollout и регламент отката |
