# ML System Design Doc - [RU]
## Дизайн ML системы - \<Автоматизаци определения ориентации железнодорожных вагонов\> \<MVP 1\>

## 1. Цели и предпосылки

### 1.1. Зачем идем в разработку продукта?

**Бизнес-цель (Product Owner)**


Стратегическая цель - повышение операционной эффективности диспетчерской службы за счет автоматизации процесса идентификации ориентации подвижного состава.


Внедрение ML-решения позволит:
- Снижение OPEX на 85% за счет автоматизации ручного труда диспетчеров
- Повышение качества мониторинга за счет снижения ошибок идентификации, что минимизирует риски аварий.
- Масштабирование без роста затрат — подключение новых полигонов без найма персонала, обеспечение 24/7 мониторинга без сменной работы.

**Почему станет лучше, чем сейчас (Product Owner & Data Scientist)**


*Текущее состояние:*
- Ручной анализ видеозаписей операторами
- Субъективность оценки при плохой видимости
- Невозможность масштабирования без увеличения штата

*Целевое состояние:*
- Автоматическая детекция референтных объектов
- Объективная оценка с доверительной вероятностью >95%
- Интеграция с существующей системой мониторинга
- Масштабируемость на любой полигон без дополнительных затрат

**Критерии успеха итерации (Product Owner)**
1. Достижение точности определения ориентации вагона ≥ 92% на валидационной выборке
2. Время отработки модели ≤ 350 мс на кадр


### 1.2. Бизнес-требования и ограничения

**Бизнес-требования**
- Система должна определять ориентацию вагона с точностью ≥90%
- Время обработки состава ≤ 5 минут
- Работа в условиях ограниченной видимости (ночь, осадки)

**Бизнес-ограничения**

2. **Нормативные:**
   - Соответствие 152-ФЗ "О персональных данных"
   - Соблюдение политики информационной безопасности заказчика

**Ожидания от итерации**
1. Рабочий прототип ML-модели

3. Отчет по результатам тестирования на валидационных данных
4. План пилотного внедрения с метриками эффективности

## 2. Методология (Data Scientist)

### 2.1. Постановка задачи

Задача решается в два этапа:
1. **Детекция объектов** — нейросеть YOLO ищет на кадре ключевые элементы вагона(тормозной цилиндр, запасной резервуар, вентиль).
2. **Логическое правило** — скрипт принимает решение: когда модель распознает объект/объекты, тогда скрипт делает вывод на основе базы правил ориентации для каждого типа вагона - правильная (A)/неправильная (B) ориентация. К примеру, если модель на камере с правой стороны вагона типа '19-752' обнаружила вентиль → скрипт сверяет в таблице правило ориентации и сохраняет результат.


### 2.2. Блок-схема решения

```
АСУ → кадры вагона → YOLO (ищет объекты)
→ Результат: найден объект / не найден → Скрипт: сверка с таблицей ориентации
→ Сохранение результата
```

### 2.3. Этапы разработки решения

#### **Этап 1: Исследовательский анализ данных (EDA)**

**Цель:** подтвердить, что предоставленных данных достаточно для обучения качественной модели.


Мы проверим:
- **Распределение сценариев:** достаточно ли в выборке кадров, снятых в разных условиях (ночь/день, снег/дождь/солнце)?
- **Анализ признаков:** существует ли устойчивая связь между положением объектов и ориентацией вагона.
- **Минимальный объем данных:** расчет необходимого количества размеченных кадров для статистической значимости результатов.

**Результат:** датасет состоит из 100 000+ изображений(кадров) разных типов вагона и ракурсов с камер. Данные разбиты по категориям - технические и погодные дефекты (затемнения/блики, дождь/снег, грязь, наклон камеры). Всего 4 камеры - с каждой стороны по 2 камеры (снизу, сверху). Разные ракурсы помогают получить полную картину логики распределения объектов для каждого типа вагона. Разрешение изображений 1280x720, формат jpg, jpeg, png, фильтры отсутствуют. Целевой минимум данных — 10 000 для статистической значимости результатов.

---

#### **Этап 2: Подготовка данных**

**Цель:** подготовить данные для обучения.

- **Формирование выборки:** берем 15000 случайных изображений с одной камеры и разделяем по принципу 70% на обучающую, 15% на валидационную и 15% на тестовую выборки.
- **Разметка данных:** размечаем 85% изображений (12750) при помощи CVAT и конвертируем в YOLO-формат.

---

#### **Этап 3: Быстрая проверка гипотезы**

**Цель:** минимально жизнеспособное решение для подтверждения, что модель вообще может решать задачу.

**Подход:** берем YOLO11n и обучаем на **маленькой размеченной выборке** (1000-1500 кадров), чтобы быстро проверить гипотезу.

**Что делаем:**
1. Берем всего 1000-1500 репрезентативных кадров (разные ракурсы, освещение)
2. Обучаем YOLO11n на этой малой выборке
3. Смотрим метрики: Accuracy детекции объектов

**Критерий прохождения этапа:** модель находит объекты с точностью **>85%** на тестовой выборке.

**Результат этапа:**
- Обученная модель (пусть и на малых данных)
- Понимание, работает ли подход в принципе
- Оценка, сколько нужно данных для целевой точности > 92%

**Риски:** минимальные — потратим несколько дней на разметку и обучение, но сэкономим недели, если гипотеза не подтвердится.

---

#### **Этап 4: Разработка Core ML решения**

**Цель:** обучить основную модель на полном объеме данных для достижения целевых метрик.

**Подход:** обучение кастомной модели детекции YOLOv12m на размеченном датасете из 15 000 кадров.

**Процесс обучения:**
- Обучаем YOLOv12m на тренировочной выборке
- После каждой эпохи проверяем качество на валидационной выборке
- Выбираем лучшую версию модели (которая показала максимальный результат на валидации)
- Финальную модель тестируем один раз на тестовой выборке (эта оценка идет в отчет)

**Аугментация данных:**
Применяем техники, имитирующие реальные условия (изменение яркости, добавление шума, имитация дождя/снега) для повышения устойчивости модели. Аугментации применяются только к тренировочной выборке.

**Целевые метрики:**
| Метрика | Цель | Где проверяем |
|---------|------|----------------|
| mAP@0.5 | ≥ 0.95 | На тестовой выборке |
| Precision | ≥ 0.95 | На тестовой выборке |
| Recall | ≥ 0.90 | На тестовой выборке |
| Время инференса | ≤ 350 мс на кадр | На тестовом окружении |

---

#### **Этап 5: Rule Engine и принятие решений**

**Цель:** трансформировать результаты детекции объектов в однозначный бизнес-вывод об ориентации вагона.

**Логика работы:**
1.  **Анализ объектов:** модель обнаруживает объекты на 3-5 изображениях и сохраняет взвешанный результат в JSON формате.
2.  **Применение правил:** существуют "жесткие" правила определения ориентации каждого типа вагона. Например, если тормозной цилиндр найден на камере с правой стороны вагона типа '19-752', то ориентация - A (правильная).


**Бизнес-валидация:** каждую неделю мы будем демонстрировать заказчику результаты работы пайплайна на сложных случаях, сверяя их с мнением экспертов, чтобы убедиться, что логика принятия решений соответствует реальным бизнес-процессам.

## 3. Подготовка пилота

### 3.1. Способ оценки пилота

**Дизайн пилота:** теневой режим (Shadow Mode)

Система подключается параллельно к работающим диспетчерам и работает "в тени":
- Диспетчеры продолжают работать как обычно
- Наша система будет брать по 3-5 изображений из базы данных, куда срезаются кадры с вагонов для анализа(система заказчика)
- Результаты системы **не влияют** на реальную работу, только записываются в лог
- В конце дня сравниваем решения системы и итоговые отчеты по ориентации

**Почему так:**
- Безопасно — ошибки системы не нарушат реальную работу
- Позволяет накопить статистику для сравнения
- Не требует остановки текущих процессов

**Схема проведения (4 недели):**
- Неделя 1-2: Сбор данных и калибровка (проверяем, что система вообще работает)
- Неделя 3: Сбор сравнительной статистики (система vs отчеты)
- Неделя 4: Анализ результатов и подготовка отчета

### 3.2. Что считаем успешным пилотом

**Главная метрика успеха:** Accuracy (доля совпадений решений системы с итоговыми отчетами) ≥ 95%

**Дополнительные метрики:**

| Метрика | Цель | Как считаем |
|---------|------|-------------|
| Accuracy | ≥ 95% | (верные ответы системы) / (все ответы) * 100% |
| False Positive Rate | < 5% | когда система сказала "есть объект", а его нет |
| False Negative Rate | < 5% | когда система не заметила объект, но он был |
| Скорость обработки | ≤ 350 мс/кадр | время от получения кадра до выдачи ответа |

**Критерий прохождения пилота:** Достижение Accuracy ≥ 95% на валидационной выборке за последнюю неделю тестирования.

### 3.3. Подготовка пилота (вычислительные ресурсы)

**Оценка затрат для пилота:**

Для YOLO12m ориентировочные требования:
- **Для обучения:** нужна видеокарта с 8-12GB VRAM (например, NVIDIA RTX 3060/3070 или в облаке)
- **Для работы (инференс):** можно на CPU, но лучше GPU для скорости

**Что нам нужно на пилот (2-3 месяца):**

**Вариант А (локальный, бесплатно):**
- Ноутбук/ПК с видеокартой (GTX 1060 и выше)
- 16 GB RAM
- 100 GB на диске для видео
- Итог: 0 ₽ (если есть свой компьютер)

**Вариант Б (облачный, если нет своего GPU):**
- Cloud GPU (Google Colab Pro или Yandex DataSphere) — ≈ 3000-5000 ₽/мес
- Хранилище для видео — ≈ 1000 ₽/мес
- **Итого на 3 месяца:** ~15 000-18 000 ₽

**На что потратим бюджет пилота:**
1. Аренда GPU для обучения (1-2 недели активно, потом редко)
2. Хранение размеченных данных
3. Оплата сервера для работы модели в теневом режиме

**Уточнение параметров:** После первого запуска baseline модели замерим реальную скорость и точность. Если окажется, что YOLO12m слишком тяжелый (медленно работает), упростим до YOLOv11m или v8n.

## 4. Внедрение в production

### 4.1. Архитектура решения

**Общая схема:**

```
АСУ "Транс-Телематика" (БД с кадрами)
        ↓
Frame Loader (выбор 3–5 кадров из БД)
        ↓
Inference Module (YOLO)
        ↓
Rule Engine
        ↓
Result Writer (запись в БД)
        ↓
Monitoring / Logs
```

**Ключевая особенность:**
Система разворачивается **on-premise на инфраструктуре заказчика** и работает напрямую с его базой данных — без использования внешних API.

**Компоненты:**

| Компонент        | Назначение                                            |
| ---------------- | ----------------------------------------------------- |
| Frame Loader     | Забирает 3–5 изображений вагона из БД заказчика       |
| Inference Module | Локальный модуль с моделью YOLO для детекции объектов |
| Rule Engine      | Применяет правила определения ориентации              |
| Result Writer    | Записывает результат обратно в БД                     |
| Monitoring       | Логирование, сбор метрик, алерты                      |

**Принцип работы:**

* АСУ уже сохраняет кадры в БД
* Наш сервис по триггеру забирает новые записи
* Обрабатывает изображения и определяет ориентацию
* Сохраняет результат в ту же или отдельную таблицу

---

### 4.2. Инфраструктура и масштабируемость

**Выбранная инфраструктура:**

* Развертывание: **on-premise (сервер заказчика)**
* Контейнеризация: Docker (опционально)
* GPU: при наличии, иначе CPU fallback

**Почему выбран on-premise:**

* Требования безопасности (нет передачи данных наружу)
* Работа с внутренней БД заказчика
* Минимизация сетевых задержек

**Масштабируемость:**

* Вертикальная: более мощный сервер / GPU
* Горизонтальная (ограниченно): запуск нескольких воркеров для параллельной обработки
* Очередь задач (например, через таблицу статусов в БД)

---

### 4.3. Требования к работе системы

**SLA:**

* Доступность: ≥ 99%
* Время обработки кадра: ≤ 350 мс

**Нагрузочные характеристики:**

| Параметр                | Значение      |
| ----------------------- | ------------- |
| Кадров на вагон         | 3–5           |
| Время обработки состава | ≤ 5 минут     |

**Режим работы:**

* Batch / near-real-time (по мере появления кадров в БД)

---

### 4.4. Безопасность системы

**Потенциальные уязвимости:**

* Доступ к БД с избыточными правами
* Поврежденные или некорректные изображения

**Меры защиты:**

* Доступ к БД только с read/write в нужные таблицы
* Валидация изображений перед инференсом
* Ограничение количества одновременно обрабатываемых задач
* Логирование всех операций

---

### 4.5. Безопасность данных

**Соответствие требованиям:**

* Соблюдение 152-ФЗ (персональные данные отсутствуют)
* Данные не покидают инфраструктуру заказчика

**Меры:**

* Работа внутри защищенного контура
* Разграничение прав доступа к БД
* Отсутствие внешних сетевых вызовов

---

### 4.6. Издержки

**Оценка затрат (в месяц):**

| Статья                     | Стоимость                                   |
| -------------------------- | ------------------------------------------- |
| Сервер (on-premise)        | 0 ₽ (используется инфраструктура заказчика) |
| Электроэнергия / поддержка | условно                                     |
| Поддержка модели           | ~20 000 – 30 000 ₽                          |

**Итого:**
**~20 000 – 30 000 ₽ / мес (в основном поддержка)**

**Преимущество:**

* Отсутствие затрат на облако
* Минимальные операционные расходы

---

### 4.7. Integration points

**Интеграция реализуется через БД заказчика:**

1. **Чтение данных:**

   * Таблица с кадрами вагонов
   * Метаданные (ID вагона, камера, время)

2. **Запись результатов:**

   * Таблица результатов ориентации

**Пример структуры результата:**

```json
{
  "wagon_id": "12345",
  "frames_used": 5,
  "detections": [
    {"class": "brake_valve", "confidence": 0.97}
  ],
  "orientation": "A",
  "confidence": 0.95,
  "timestamp": "2026-03-26T12:00:00"
}
```

---

### 4.8. Риски

| Риск                      | Описание                           | Митигирование                         |
| ------------------------- | ---------------------------------- | ------------------------------------- |
| Нагрузка на БД            | Частые запросы могут влиять на АСУ | Батчинг, индексы, ограничение частоты |
| Некачественные кадры      | Размытые / закрытые объекты        | Фильтрация + дообучение               |
| Медленный инференс на CPU | Нет GPU на сервере                 | Оптимизация модели / lighter версия   |
| Drift данных              | Новые условия съемки               | Мониторинг + периодический retraining |
| Ошибки в правилах         | Некорректная логика                | Валидация с экспертами                |
